### 2. Fase 1 — Correcciones CRÍTICAS

#### 2.1 — Fix LCP: eliminar opacity:0 del SSR
El `motion.div` que envolvía el widget del mailbox inyectaba `style="opacity:0"` en el HTML del servidor, bloqueando el LCP 1.5-2s.

**Solución:** Reemplazado por `<div className="animate-mailbox-in">` con CSS `animation-fill-mode: both` en `globals.css`. El HTML estático ya no contiene ningún `opacity:0` inline.

#### 2.2 — Canonicals en subpáginas
`/es/privacy`, `/es/terms`, `/en/privacy`, `/en/terms` tenían el canonical apuntando a la homepage → Google las trataba como duplicados.

**Solución:** `generateMetadata` actualizado en cada subpágina con canonical correcto + hreflang `es`, `en`, `x-default`.

#### 2.3 — Eliminación de FAQPage JSON-LD
Google restringió los rich results de FAQPage a sitios gubernamentales y de salud desde agosto 2023. El bloque era peso innecesario sin beneficio.

**Solución:** Eliminado de `FAQAccordion.tsx`.

#### 2.4 — Contenido visible en homepage (~320 palabras)
Solo ~99 palabras indexables en el HTML estático. Todo el contenido útil estaba en el FAQ colapsado con JS.

**Solución:** Añadidas 3 secciones semánticas después del widget de email:
- ¿Qué es TmpMail? (~100 palabras)
- Cómo funciona TmpMail (~110 palabras)
- Por qué usar un correo temporal desechable (~110 palabras)

Copy en ES + EN añadido a `messages/es.json` y `messages/en.json`.

#### 2.5 — ISR revalidation
`export const revalidate = 3600` añadido al layout del locale para cachear el HTML y mejorar TTFB del servidor origen.

#### 2.6 — Documentación de infraestructura
- `docs/infra/cloudflare-setup.md` — guía paso a paso para activar CDN
- `docs/infra/nginx-security.conf` — config nginx con HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy, CSP (Report-Only)

### 3. Fase 2 — Mejoras de ALTA prioridad

#### 3.1 — Eliminación completa de Framer Motion (-172 KB)
`framer-motion` era el tercer paquete más pesado del bundle (172 KB sin comprimir).

**Migraciones:**
- `FAQAccordion.tsx` → `<details>/<summary>` HTML nativo. Sin JS, accesible, Google indexa el contenido abierto.
- `QRModal.tsx` → CSS `@keyframes modalFadeIn` + `modalPanelIn`
- `package.json` → `npm uninstall framer-motion`

#### 3.2 — Optimización de Lucide React (~-175 KB)
Una línea en `next.config.js`:
```js
experimental: { optimizePackageImports: ["lucide-react"] }
```
Next.js realiza tree-shaking automático sin cambiar ningún import en componentes.

**Ahorro total de bundle: ~347 KB** de JavaScript eliminado.

#### 3.3 — Redirect raíz 301 permanente
El redirect `/` → `/es` era un 307 temporal basado en cookie `NEXT_LOCALE`. Google no transfiere link equity con 307.

**Solución:** `middleware.ts` reescrito para interceptar `pathname === "/"` **antes** de que `createMiddleware` de `next-intl` lo procese, retornando 301 directo a `/en`.

#### 3.4 — Font sizes mínimos (12px)
Google considera 12px el mínimo para texto legible en mobile. Se encontraron 37+ instancias de `text-[10px]` y `text-[11px]` en 14 archivos.

**Solución:** Bulk replace `text-[10px]` → `text-xs` y `text-[11px]` → `text-xs` (12px) en todos los componentes.

#### 3.5 — Schemas mejorados

**WebApplication** (en `page.tsx`):
- Cambiado de `SoftwareApplication` a `WebApplication`
- Añadidos: `@id`, URL locale-aware, description locale-aware, `featureList` como array, `image`, `publisher` con referencia `@id`

**Organization** (en `layout.tsx`):
- Añadidos: `@id`, logo 512×512, email en `contactPoint`

**WebSite** (en `layout.tsx`):
- Añadidos: `@id`, `inLanguage` array `["es", "en"]`, `publisher` con referencia

#### 3.6 — FAQ como HTML nativo indexable
Las respuestas del FAQ estaban colapsadas en JS — Google podría no indexarlas. Con `<details>/<summary>` el texto está en el DOM desde el primer render. Suma ~180 palabras adicionales indexables (total homepage: 500+).

#### 3.7 — Auditoría Socket.IO
Documentado en `docs/infra/socketio-audit.md`. Conclusión: mantener Socket.IO — usa rooms, auto-reconnect y eventos bidireccionales que requieren lógica custom si se migra a WebSocket nativo. Re-evaluar en Fase 4.

### 5. Infraestructura aplicada en servidor

#### 5.1 — HTTP/2 activado
```nginx
# /etc/nginx/sites-enabled/tmpmailio.com
listen [::]:443 ssl http2 ipv6only=on;
listen 443 ssl http2;
```
Verificado: `HTTP/2 200` ✅

#### 5.2 — Security headers (`tmpmailio-security.conf`)
Aplicados en `/etc/nginx/conf.d/tmpmailio-security.conf`:
- `Strict-Transport-Security` (HSTS, 1 año, includeSubDomains, preload-ready)
- `X-Frame-Options: DENY`
- `X-Content-Type-Options: nosniff`
- `Referrer-Policy: strict-origin-when-cross-origin`
- `Permissions-Policy: camera=(), microphone=(), geolocation=()`
- `Content-Security-Policy-Report-Only` (modo auditoría, sin bloquear)
- `server_tokens off` + `poweredByHeader: false` en Next.js

#### 5.3 — Cloudflare CDN configurado
- Nameservers delegados a Cloudflare
- Registro A proxied (`187.77.234.184`)
- SSL/TLS: Full (strict)
- Cache Rules: Cache Everything + respetar `Cache-Control` de Next.js
- Brotli: habilitado
- Registros DKIM corregidos a DNS only (evita rotura de correo Hostinger)

### Paso 3 — Páginas de uso específico (mayor ROI de contenido)

Crear páginas tipo `/es/[slug]` (~600 palabras cada una) enfocadas en casos de uso concretos. Tienen menos competencia que keywords genéricas:

**URLs propuestas:**
```

/es/email-desechable-sin-registro
/es/proteger-privacidad-email
/es/correo-temporal-redes-sociales
/es/evitar-spam-correo-personal
```

**Estructura de cada página:**
1. H1 con keyword principal
2. Párrafo intro (~100 palabras)
3. Sección "Cómo usar TmpMail para X" con pasos numerados
4. Sección "Por qué TmpMail es la mejor opción para X"
5. FAQ específico de ese caso de uso (2-3 preguntas con `<details>`)
6. CTA → link a la app con el widget de generación
